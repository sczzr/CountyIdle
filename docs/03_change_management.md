# CountyIdle 大纲优化与改动流程

## 1. 什么时候允许改大纲

- 玩家体验目标变化（如从“轻挂机”转向“重策略”）
- 当前版本数据证明核心飞轮失衡
- 技术约束导致原方案不可实现

不满足以上条件时，优先在参数层优化，不改核心方向。

## 2. 标准改动流程

1. 写提案：使用 `templates/change-proposal-template.md`
2. 做影响评估：影响哪些系统、是否破坏飞轮
3. 小步上线：只改一个主要目标
4. 记录结果：使用 `templates/balance-log-template.md`
5. 决策归档：继续、回滚或二次迭代

## 3. 变更分级

- L1（参数微调）：掉率、系数、冷却时间等
- L2（机制增强）：新增词条、事件、建筑效果
- L3（架构级变化）：重排核心循环或系统依赖

建议：
- 单迭代最多 1 个 L2 或 L3 项
- L3 改动必须先出提案 + 验收指标

## 4. 决策标准（每次都要回答）

- 这次改动是否提升“可挂机性 + 可决策性”？
- 这次改动是否增加了无效复杂度？
- 玩家 10 分钟内能否感知收益？

## 5. 回滚规则

- 若核心指标连续 2 个评估周期恶化，回滚到上一个稳定版本。
- 回滚后必须写明失败原因，避免重复试错。
