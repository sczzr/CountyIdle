# CountyIdle 变更立卷法

> 本文规定 CountyIdle 的正式变更如何入门、如何立卷、如何验收、如何回写。  
> 它不裁决项目愿景，那是 [01_game_design_guide.md](/E:/2_Personal/Finally/docs/01_game_design_guide.md) 的职责；  
> 它不沉淀系统公式，那是 [02_system_specs.md](/E:/2_Personal/Finally/docs/02_system_specs.md) 的职责。  
> 本文只管一件事：**既然决定要改，就必须按门规改。**

## 1. 文档地位与判案顺序

凡涉及以下任一事项，皆属正式变更：

- 机制与数值
- 世界观、术语与命名
- UI 与表现层结构
- 主循环、时间、存档与基础架构
- 功能立项、续做、停用与恢复

所有正式变更，必须按以下顺序对齐：

1. [09_xianxia_sect_setting.md](/E:/2_Personal/Finally/docs/09_xianxia_sect_setting.md)
2. [01_game_design_guide.md](/E:/2_Personal/Finally/docs/01_game_design_guide.md)
3. [02_system_specs.md](/E:/2_Personal/Finally/docs/02_system_specs.md)
4. [05_feature_inventory.md](/E:/2_Personal/Finally/docs/05_feature_inventory.md)
5. [08_development_list.md](/E:/2_Personal/Finally/docs/08_development_list.md)
6. `FC / CP / BL` 过程文档

裁定：

- 禁止跳过文档直接编码。
- 禁止先改实现、再倒推主法。
- 若上位文档与下位文档冲突，一律以上位文档为准。

## 2. 受理门槛

每次收到需求，先做三步判定：

1. 先查 [05_feature_inventory.md](/E:/2_Personal/Finally/docs/05_feature_inventory.md)
2. 再查 [08_development_list.md](/E:/2_Personal/Finally/docs/08_development_list.md)
3. 最后判它属于哪一档

三档裁定如下：

- `✅ 已实现`：按优化、修复或重构受理，不重复造功能。
- `🟡 部分实现`：按续做受理，沿现有功能包继续推进。
- `⭕ 未实现`：先登记开发列表，再进入设计与实现。

当需求只写“继续完善游戏”而未指定条目时，默认按 `08` 的顺序逐条推进。  
当需求与 `01` 的主线无关时，不入主开发链。

## 3. 变更分级

### 3.1 L1 微调

适用范围：

- 掉率、系数、冷却、产量、消耗
- 文案修正、提示优化、轻度布局整理
- 不改系统边界的体验修补

门规：

- 必须记录改前 / 改后。
- 必须写清预期收益与潜在副作用。
- 不得把多项独立问题打包伪装成一次微调。

### 3.2 L2 机制增强

适用范围：

- 新增事件、条目、建筑效果、分支选择
- 外务、治理、成长回路扩展
- 对既有系统新增判断空间的改动

门规：

- 必须补功能卡。
- 若影响平衡、节奏或飞轮，必须补提案。
- 必须回写 `02 / 05 / 08`。

### 3.3 L3 架构级改动

适用范围：

- 主循环重排
- 数据结构与系统依赖重构
- 存档格式、结算链、基础时间口径调整

门规：

- 必须先出提案与验收指标。
- 必须明确回滚路径。
- 单轮迭代不并行推进多个 `L3` 主任务。

## 4. 立卷规则

正式改动一律按卷宗留痕：

- 功能卡：`FC-日期-topic`
- 变更提案：`CP-日期-topic`
- 平衡日志：`BL-日期-topic`

立卷裁定：

- 同一条改动链必须复用同一 `topic`。
- 同义主题不得反复新建卷宗。
- 命名与归并规范以 [06_archive_registry.md](/E:/2_Personal/Finally/docs/06_archive_registry.md) 为准。

何时必须立卷：

- 新功能立项：必立 `FC`
- 机制增强或架构改动：必立 `FC + CP`
- 数值调整且影响节奏、收益或飞轮：必立 `BL`

## 5. 标准闭环

任何正式改动，都按以下顺序推进：

1. 判它服务哪一条主线或哪一段飞轮
2. 判它在 `05 / 08` 中属于优化、续做还是新项
3. 立卷并写明完成定义
4. 若涉及机制、平衡、架构，先回写 `01 / 02`
5. 做最小闭环实现：`models -> systems -> core -> UI`
6. 做最低验证
7. 回写 `02 / 05 / 08` 与归档文档

裁定：

- 不允许只做半截实现而不改状态文档。
- 不允许只改文档、不说明实现入口。
- 不允许用局部完成冒充整项完成。

## 6. 立项前六问

每次改动前，至少要答清以下六问：

- 这次改动强化的是宗门运营、弟子调度，还是挂机自动运转
- 这次改动是否让天衍峰更像一个活宗门
- 这次改动是否增加了新的判断重量，而不是新的操作噪音
- 玩家是否能在合理时长内感知收益与代价
- 新文案是否仍有旧县城 / 郡县语义漏出
- 这次改动是否会伤及小时结算、存档或主循环稳定性

若六问答不清楚，不得进入正式开发。

## 7. 最低验证与结案

每次改动至少应覆盖以下验证：

- `dotnet build .\Finally.sln`
- Godot 运行烟测
- 小时结算节奏是否正确
- 人口 / 职司 / 资源是否出现负值或超分配
- 存档 / 读档是否保持基本可用

结案裁定：

- 未完成验证时，必须明确标注“未完成项”。
- 禁止把“未验证”写成“已完成”。
- 若验证只覆盖局部，结论只能写“部分成立”。

## 8. 回滚法

满足以下任一条件时，应直接进入回滚评估：

- 核心指标连续恶化
- 主法与当前实现明显冲突
- 新改动破坏存档兼容
- 新改动破坏主循环稳定性或可读性
- 表现层改动遮蔽了系统判断与反馈

回滚后必须留下三项记录：

- 失败原因
- 受影响范围
- 下一轮可执行修正方向

## 9. 禁行事项

- 未登记就插入零散功能
- 跳过飞轮映射直接编码
- 用局部实现反向篡改主法口径
- 在新文档里重新引入“郡县经营”主设定
- 以草案身份冒充当前版本既成事实
- 把表现层流水、施工残迹、迁移口述写进正式主文
