# CountyIdle 宗门玩法改动流程

## 1. 什么时候允许改总纲或世界观文档

满足以下任一条件时，允许改 `09 / 01 / 02` 这一层文档：

- 玩家体验目标变化，例如从“轻挂机宗门经营”转向“更强策略管理”
- 当前版本数据证明核心飞轮失衡
- 旧文档残留县城 / 郡县语义，已经影响新功能理解与实现
- 技术约束导致原方案不可实现，必须重新定义规则边界

如果不满足以上条件，优先做局部参数或表现层优化，不轻易改大方向。

## 2. 文档对齐顺序（强制）

涉及机制、世界观、命名或边界的任务，必须按以下顺序处理：

1. `09_xianxia_sect_setting.md`：先确认宗门设定和术语
2. `01_game_design_guide.md`：再确认飞轮目标与设计边界
3. `02_system_specs.md`：最后确认公式、顺序、系统规则
4. `05_feature_inventory.md` / `08_development_list.md`：回写状态和顺序
5. `FC / CP / BL`：补齐过程文档

禁止跳过文档直接编码。

## 3. 标准改动流程

1. 先判定需求属于哪个飞轮环节
2. 查 `05_feature_inventory.md` 确认是已实现、续做还是新项
3. 若是新项，先登记 `08_development_list.md`
4. 写功能卡：使用 `templates/feature-card-template.md`
5. 若涉及机制 / 平衡 / 架构，再补提案：`templates/change-proposal-template.md`
6. 做最小闭环实现：`models -> systems -> core -> UI`
7. 执行最低验证：`dotnet build .\Finally.sln` + Godot 烟测
8. 写结果记录：`templates/balance-log-template.md`
9. 回写 `02 / 05 / 08` 与相关归档

## 4. 变更分级

- L1（参数微调）：掉率、系数、冷却、产量、消耗、显示文案微调
- L2（机制增强）：新增词条、事件、建筑效果、分支选择、外务行为
- L3（架构级变化）：重排核心循环、数据结构、系统依赖、存档格式

建议：

- 单迭代最多同时推进 `1` 个 L2 或 L3 主任务
- L3 改动必须先出提案与验收指标
- 如果是“文档重梳但不改规则”，至少落功能卡与结果记录

## 5. 决策标准（每次都要回答）

- 这次改动是否增强了“天衍峰经营感”？
- 这次改动是否提升了“可挂机性 + 可决策性”？
- 这次改动是否增加了无效复杂度？
- 玩家是否能在 `10` 分钟内感知收益？
- 新文案是否仍有旧县城 / 郡县语义漏出？

## 6. 回滚规则

- 若核心指标连续 `2` 个评估周期恶化，回滚到上一个稳定版本
- 若新文档与现行代码 / UI 语义冲突，优先回滚错误文档描述或补兼容说明
- 回滚后必须写明失败原因，避免重复试错

## 7. 归档命名与去重规则

- 统一使用 `FC / CP / BL + 日期 + canonical-topic[-NN]` 命名
- 同一改动链路（功能卡 / 提案 / 日志）必须复用同一个 `canonical-topic`
- 同义主题不重复建新 topic
- 默认不加 `-NN`；仅在同目录同日同 topic 产生重名时添加序号
- 历史归并与 topic 规范以 `06_archive_registry.md` 为准

## 8. 新需求受理协议（开发前置）

每次收到新需求，按以下顺序执行：

1. 先查 `05_feature_inventory.md`
   - `✅ 已实现`：进入优化 / 修复流
   - `🟡 部分实现` 或 `⭕ TODO`：直接续做对应功能
2. 若未命中任何条目：先登记到 `08_development_list.md`，形成完整功能包后再开发
3. “继续完善游戏”默认按 `08_development_list.md` 顺序逐条推进

## 9. 禁止行为

- 未登记就插入零散功能
- 跳过飞轮映射直接编码
- 只做半截功能且不回写状态文档
- 在新增文档里重新引入“郡县经营”主设定
