# 功能卡（Feature Card）

## 功能信息

- 功能名：岗位面板具体岗位化与进程联动
- 优先级：`P1`
- 目标版本：`v0.x`
- 关联系统：`GameState / JobRoleSystem / GameLoop / JobsPanel / JobRow / jobs.json`

## 目标与用户收益

- 目标：让 `JobsPadding` 从“四个笼统岗位”升级为“与建筑、科技联动的具体岗位列表”，并支持逐项交互。
- 玩家可感知收益（10 分钟内）：玩家能直接看到“建了什么建筑、突破了哪段科技、因此多了哪些岗位”，并能按具体岗位而非大类调配人数。

## 实现范围

- 包含：
  - `jobs.json` 接入具体岗位定义与解锁规则
  - 左侧岗位面板改为动态生成具体岗位条目
  - 每个条目支持点击查看规则、`+/-` 调配、优先保留切换
  - 具体岗位分配自动汇总回四大岗位总量，保持现有经济结算兼容
  - 旧存档自动补齐具体岗位分配
- 不包含：
  - 分支科技树本体
  - 每个具体岗位单独改写资源产出公式
  - Figma 布局岗位面板重做

## 实现拆解

1. 补齐系统规格、开发列表与改动提案
2. 新增具体岗位配置模型与规则系统
3. 改造 `GameState / GameLoop`，接入具体岗位分配同步
4. 重写 `JobsPadding` 为动态岗位条目 UI
5. 构建验证并补平衡日志

## 验收标准（可测试）

- [ ] `JobsPadding` 中每个岗位元素都可交互，并能显示具体岗位名称
- [ ] 岗位加减受建筑数量、科技阶段、空闲人口与大类容量共同约束
- [ ] 锁定岗位能显示明确解锁条件，优先保留岗位能在回退时被最后裁减
- [ ] 读档后具体岗位分配可恢复，四大岗位总量与经济结算保持兼容
- [ ] `dotnet build .\Finally.sln` 通过

## 风险与回滚

- 风险：具体岗位数量增加后，面板信息密度显著上升，若说明不够清楚会提升理解成本。
- 回滚方式：保留 `jobs.json` 与规则系统，岗位面板临时退回四大岗位汇总视图，仅保留规则数据层。
