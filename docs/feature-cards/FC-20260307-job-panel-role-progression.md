# 功能卡（Feature Card）

## 功能信息

- 功能名：岗位面板进程化交互
- 优先级：`P1`
- 目标版本：`v0.x`
- 关联系统：`Industry / Research / MainUI`

## 目标与用户收益

- 目标：让 `JobsPadding` 中的岗位行既可交互查看规则，又能以建筑/科技进程驱动具体岗位名称。
- 玩家可感知收益（10 分钟内）：点开岗位行即可看懂容量来源、当前具体岗位身份与下一步解锁方向，不再面对抽象“农工岗/学士岗”。

## 实现范围

- 包含：
  - 新增岗位进程规则，按建筑数量与 `TechLevel` 决定当前岗位名称
  - `JobsPadding` 每个岗位行支持展开/收起详情
  - 岗位详情展示容量公式、当前建筑快照、已派人数、下一阶解锁条件
  - 岗位调整与回退日志改用当前具体岗位名称
- 不包含：
  - 将四大职业拆分为独立可分配的子职业数值池
  - `jobs.json` 的完整配置化读取
  - Figma 布局岗位面板重做

## 实现拆解

1. 先补系统规格、开发列表、提案与平衡日志记录入口
2. 新增岗位进程规则层，生成岗位标题/摘要/详情
3. 接入 Legacy `JobsPadding` UI 交互并同步日志文案

## 验收标准（可测试）

- [x] `JobsPadding` 四条岗位行都能点击展开/收起详情
- [x] 岗位标题可随建筑/科技进度切换为具体岗位名称
- [x] 详情文本能解释容量公式、建筑快照与下一阶解锁条件
- [x] 岗位调整/超编日志使用当前具体岗位名称
- [ ] Godot 内手点验证 Legacy 岗位面板视觉表现
- [x] `dotnet build .\Finally.sln` 通过

## 风险与回滚

- 风险：岗位名称切换过于频繁时，可能造成玩家对当前抽象职业含义的认知波动。
- 回滚方式：移除岗位进程规则层，恢复原有固定岗位标题与静态行布局。
