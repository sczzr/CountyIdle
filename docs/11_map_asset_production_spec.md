# CountyIdle 地图素材生产规格（v0.1 草案）

> 用途：统一天衍峰山门图与后续世界地图的视觉基调、素材分层、命名、尺寸、锚点、拼接逻辑与交付口径，作为地图表现层的正式生产规范。

## 1) 适用范围

- 天衍峰山门图（当前主线地图）
- 世界地图与后续二级地图的基础地貌表现
- 宗门图 / 世界图的道路、河流、法阵、奇观、氛围层
- 美术出图、程序接入、后续验收与替换

不包含：

- 小头像、UI icon、按钮图标
- 弟子立绘与角色动画
- 实时光照或 3D 场景生产规范

## 2) 设计目标

- 保持“青绿山水 + 工笔重彩 + 羊皮古卷”的统一气质。
- 让地图既像一幅被画在卷轴上的修仙古图，又保留清晰的格位识别与玩法可读性。
- 为当前 `pointy-top` 六边格地图提供可复用、可扩展、可程序拼接的资产体系。
- 优先服务 `地图点击 -> 左栏检视 -> 推荐动作` 主链路，不让表现层破坏交互可读性。

## 3) 视觉基调

### 3.1 风格关键词

- 青绿山水
- 工笔重彩
- 羊皮古卷
- 毛笔勾线
- 宣纸晕染
- 做旧水渍

### 3.2 色彩原则

- 禁用高饱和西式奇幻霓虹配色。
- 颜色优先取自中国传统色系，强调沉着、矿物感、纸面感。
- 单张图的高亮色面积必须受控，避免全图同时发光。

推荐基色：

| 主题 | 推荐色名 | 用途 |
| --- | --- | --- |
| 灵山 / 正道 / 高灵气地带 | `石青`、`石绿`、`螺青` | 山脉、宗门、灵脉、仙山 |
| 凡俗聚落 / 土地 / 荒地 | `赭石`、`米黄`、`雁灰` | 平原、城邑、道路、低山 |
| 水域 / 深潭 / 归墟 | `花青`、`黛蓝`、`青碧` | 河流、湖海、浅滩、深水 |
| 秋林 / 灵草 / 稻田 | `藤黄`、`秋香`、`竹青` | 林地、田地、草木点缀 |
| 危险地貌 / 瘴气 / 妖氛 | `乌青`、`暗绿`、`紫黯` | 毒沼、妖城、鬼林、腐化地 |
| 火脉 / 炼器 / 炎地 | `朱砂`、`赤金`、`焦褐` | 火山、炉脉、法阵高亮 |

### 3.3 质感原则

- 地图最底层始终建立在卷轴 / 羊皮纸底材之上。
- 所有地块、道路、河流、物件都应体现“被画在纸面上”的质感，而不是漂浮贴上去的现代 UI 图块。
- 边缘过渡优先使用淡墨、毛刷、晕染，而不是机械硬裁边。
- 允许保留少量纸张纤维、老化斑点、水痕，但不能抢前景交互。

## 4) 技术基线

### 4.1 视角与网格

- 视角：2.5D 固定等距视角
- 六边形朝向：`pointy-topped`
- 当前逻辑网格：沿用 `TownMapData(width, height)` 与现有世界图坐标系统
- 当前主线地图：天衍峰山门图

### 4.2 总体原则

- 地图不是一整张大画，而是“卷轴底图 + 六边格内地形 + 可拼接线路 + 可破框立体物件 + 顶层氛围”的分层拼装系统。
- `Layer 1 / Layer 2` 必须 tile 化，便于逻辑驱动与程序拼接。
- `Layer 3 / Layer 4` 允许半自由摆放，但仍需遵守锚点与遮挡规则。
- 不使用引擎实时光照作为主表达方式，光影直接预绘在素材内。

## 5) 分层结构（强制）

### 5.1 Layer 0：卷轴底图

职责：

- 提供整张地图的纸面、卷轴、做旧和轻微水渍基底。
- 作为地图整体氛围来源，不承担玩法语义。

规范：

- 全屏静态 UI 层。
- 不参与六边格对齐。
- 优先使用可裁切或可 cover 的长画幅底图。
- 对比度应低于前景地块与物件，避免抢占注意力。

### 5.2 Layer 1：基础地块

职责：

- 表达地貌底盘，是所有地图语义的第一层。

规范：

- 必须严格限制在六边形边界内。
- 必须支持多变体，以降低平铺重复感。
- 允许边缘使用淡墨、毛刷、纸面晕染做软过渡，但中心主体不得越界。
- 不承载高物件，不承担明显遮挡关系。

### 5.3 Layer 2：道路 / 河流 / 法阵图案

职责：

- 表达连接关系、路径引导和地表图案。

规范：

- 必须 tile 化并可按六方向逻辑拼接。
- 允许轻微越过格边，但不得破坏地块识别。
- 道路、河流、法阵均视作“连接类资产”，优先复用同一套连接逻辑。

### 5.4 Layer 3：立体物件层

职责：

- 表达山峰、建筑、树木、奇观、节点与资源物件。

规范：

- 开启 `Y-Sorting` 或等价的按 Y 轴排序逻辑。
- 物件底部锚点必须落在所属 hex 中心附近或该格底盘定义位置。
- 允许顶部突破六边形上半边界，即“破框”。
- 破框只允许发生在高物件，不允许基础地块整体越框。

### 5.5 Layer 4：氛围与特效层

职责：

- 柔化网格边缘，提升卷轴仙气与动态感。

规范：

- 不必严格对齐六边格。
- 可漂浮、缓动或固定在特定区域。
- 低缩放时应允许淡出或减弱，避免覆盖交互信息。

## 7) 连接类拼接规则（强制）

### 7.1 逻辑表达

- 每个 hex 的连接类资产按六方向记录连通关系。
- 推荐使用“六方向 bitmask + 样式集”表达道路、河流、法阵。
- 同一逻辑可复用于多种视觉主题，只替换样式集，不替换连接规则。

### 7.2 最低拼接组合

每套连接类资产至少提供以下组合：

- 单边连接
- 对边直连
- 相邻双边转角
- 三边分叉
- 四边复合
- 五边复合
- 六边枢纽
- 无连接中心图

### 7.3 视觉要求

- 河流优先表现“水从中心向边缘流出”的连续感，不允许出现突然断裂的机械接缝。
- 道路应保留笔刷或砂土肌理，不能像现代柏油路。
- 法阵纹路可更规整，但仍应保留卷轴绘制感。

## 8) 破框遮挡规则（核心）

### 8.1 正确原则

- 物件底座符合 hex 占地。
- 物件上半部分可突破本格上缘，覆盖上方一行格子的下半区域。
- 山、塔、古树、城楼、奇观是主要破框对象。

### 8.2 锚点规则

- 立体物件的主锚点位于其所属格的底部中心或近中心偏下位置。
- 视觉重心允许上移，但命中与所属逻辑仍归属锚点所在格。

### 8.3 控制规则

- 同一屏强破框物件数量应受控，避免大面积遮挡。
- 基础地块禁止破框。
- 道路 / 河流只允许轻微越界，不允许大面积覆盖相邻格。

## 9) 光影规范

- 不启用实时主光源作为核心表现。
- 假定主光永远从左上角照射。
- 山脉、建筑、树木的亮面在左上，阴影画在右下。
- 物件底部可带半透明软阴影，帮助其压入纸面与地块。
- 所有阴影都应表现为预绘质感，不做现代硬边投影。

## 10) 网格线规范

- 禁止使用纯黑 1px 实线六边形网格。
- 推荐使用低透明暗金线或断续墨线。
- 默认网格存在感要弱，只在玩家需要辨识格位时提供辅助。
- 选中态高亮与基础网格线应明确分开，不能混用同一视觉语言。

## 11) 尺寸、锚点与画布规范

### 11.1 基础原则

- 同一类资产的画布尺寸必须统一。
- 逻辑占地、画布范围、锚点位置必须分开定义。
- 允许“画布大于占地”，但不允许锚点漂移。

### 11.2 推荐约束

- `Layer 1` 基础地块：统一六边格裁切尺寸，严格在格内
- `Layer 2` 连接类：统一与基础地块同格尺寸，必要时允许少量出血边
- `Layer 3` 单格立体物件：画布高度可为单格的 `1.5x ~ 2.5x`
- `Layer 3` 多格奇观：必须额外定义占地格集合与主锚点
- `Layer 4` 氛围资产：自由尺寸，但需标注推荐覆盖范围

### 11.3 锚点命名建议

- `center`
- `bottom_center`
- `footprint_center`
- `effect_center`

立体物件默认使用 `bottom_center`。

## 12) 命名与目录规范

### 12.1 目录建议

```text
CountyIdle/assets/map/
  scroll/
  terrain/
  connectors/
    roads/
    rivers/
    arrays/
  props/
    mountains/
    vegetation/
    poi/
    wonders/
  atmospherics/
  manifests/
```

### 12.2 命名规则

统一格式：

```text
map_{layer}_{category}_{name}_v{variant}
```

示例：

- `map_l1_terrain_plain_v01`
- `map_l1_terrain_spirit_vein_v02`
- `map_l2_river_corner_v01`
- `map_l3_mountain_immortal_peak_v03`
- `map_l3_poi_market_pier_v01`
- `map_l4_cloud_auspicious_v02`

### 12.3 配套清单

每批资产应至少附带一份 manifest，包含：

- `asset_id`
- `layer`
- `category`
- `variant`
- `footprint`
- `anchor`
- `suggested_biomes`
- `notes`

## 14) 程序接入边界

### 14.1 当前系统约束

- 当前宗门图仍以程序化 hex 绘制与全格检视为主，不直接依赖完整贴图流水线。
- 本文档是后续替换或补强 `Layer 1~4` 的生产规范，不要求一次性推翻现有渲染逻辑。
- `Layer 1 / Layer 2` 优先设计成可被 `TownMapGeneratorSystem` 或后续世界图生成器直接消费的资产集合。
- 运行时一期建议先走“`Layer 1 atlas + Layer 2 decal / connector + terrain 语义生成`”路线，先打通 Godot 贴图接入，再逐步替换为正式量产素材。

### 14.2 接入建议顺序

1. 先接 `Layer 1` 基础地块
2. 再接 `Layer 2` 道路 / 河流 / 法阵
3. 再接 `Layer 3` 高辨识度立体物件
4. 最后接 `Layer 4` 氛围层

### 14.3 不应发生的事情

- 不应让单张手绘大图覆盖所有逻辑格位，导致无法复用。
- 不应把玩法信息全部塞进氛围素材里。
- 不应让氛围层遮挡点击、选中与检视反馈。
- 不应为了画面完整性破坏现有 `tile 选中 -> 检视器刷新` 主链路。

## 15) 验收清单

- [ ] 视觉上能稳定读出“卷轴古图”气质，而不是通用奇幻地表。
- [ ] 玩家能在低缩放下分清基础地貌、路径与重点节点。
- [ ] 山脉 / 建筑 / 奇观的破框遮挡不影响所属格判定。
- [ ] 道路 / 河流 / 法阵可以按逻辑拼接，不依赖手工逐格摆图。
- [ ] 资产目录、命名、变体数、锚点信息明确，后续可批量扩充。
- [ ] 低缩放状态下，氛围层不会遮住网格与选中反馈。

## 16) 与现有主线的关系

- 本文档服务于“宗门 hex 驻地地图系统”与“天衍峰院域坊局系统”的表现层升级。
- 本文档是 `docs/02_system_specs.md` 的配套生产规格，不替代玩法公式与交互链路。
- 若后续世界格二级地图进入运行时实现，应优先复用本规范中的分层、命名与拼接规则，而不是另起一套表现标准。
