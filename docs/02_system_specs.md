# CountyIdle 系统规格（初版）

> 用于统一“设计语言”，避免只写概念不落地。

## A. 郡城与人口系统

### 目标
- 提供稳定劳动力与增长曲线。

### 输入
- 粮食结余、幸福度、人口上限、外部威胁。

### 输出
- 人口变化、生产效率系数、流失/怠工事件。

### 关键规则（待填实数）
- 人口增长率 = `基础增长率 × 幸福度系数 × 粮食系数`
- 幸福度影响生产效率：`效率 = f(幸福度)`

### 边界条件
- 粮食不足时触发负增长或幸福下降。
- 幸福低于警戒线时触发流失。

## B. 职业与科技系统（士农工商）

### 目标
- 让玩家在生存、发展、科研之间做分配权衡。

### 输入
- 总人口、职业分配比例、资源库存。

### 输出
- 粮木石、金币、科研点、装备相关产能。

### 关键规则（待填）
- 农：基础资源产出公式
- 工：加工/锻造产出公式
- 商：资源转金币与幸福增益公式
- 士：科研点产出与科技解锁条件

## C. 精英繁育与血脉系统

### 目标
- 提供长期养成与“抽卡式惊喜”。

### 输入
- 父母属性、血缘关系、环境/建筑加成。

### 输出
- 后代八维属性、天赋词条、突变事件。

### 关键规则（待填）
- 子代属性 = `父母均值 + 随机波动`
- 近亲惩罚函数
- 突变概率与上限保护

## D. 装备与锻造系统

### 目标
- 构建中后期成长驱动力。

### 输入
- 图纸、材料、铁匠灵巧与天赋。

### 输出
- 品质、词条、耐久、熔炼产物。

### 关键规则（待填）
- 品质概率池（白绿蓝紫金）
- 词条池权重
- 耐久消耗与修复/熔炼回收比例

## E. 探险与守城系统

### 目标
- 用自动战斗输出稀缺资源，并对后方经营施压。

### 输入
- 英雄编队、装备、补给、当前层数、城防值。

### 输出
- 掉落、战损、层数推进、攻城事件结果。

### 关键规则（待填）
- 每层难度增长曲线
- Boss 机制与词条克制
- 攻城频率与强度计算

## F. 郡学突破（科研里程碑）系统

### 目标
- 将“士”职业产出的科研点转化为可见的中期增益，强化“职业分化 → 长线成长”反馈。

### 输入
- `Research`（科研点累计）
- 当前科技阶 `TechLevel`

### 输出
- 生产加成倍率（粮食/工产/贸易）
- 人口增长倍率
- 科技突破日志（用于玩家可观测反馈）

### 关键规则（v0.x）
- 每小时结算后根据 `Research` 判定可达科技阶：
  - T1（30）：粮食产量倍率 `1.15`
  - T2（90）：木石产量倍率 `1.15`，金币产量倍率 `1.10`
  - T3（180）：人口增长倍率 `1.20`
- 科技阶只升不降（同一存档内保持长期成长）。
- 产出公式示例：
  - `FoodGain = Farmers × BaseFood × FoodProductionMultiplier`
  - `GrowthRate = BaseGrowthRate × ... × PopulationGrowthMultiplier`

### 边界条件
- 默认倍率必须为 `1.0`，避免读档旧数据时出现 0 倍率。
- 科技倍率不可为负；若状态异常，结算时按最低 `1.0` 处理。

## G. 郡县见闻（动态事件）系统

### 目标
- 提供每小时可感知的“郡务波动”，把职业分化与探险状态转为短周期反馈。

### 输入
- `Merchants / Scholars / ElitePopulation`
- `Happiness / Threat / ExplorationDepth`
- 资源库存（粮食、金币、科研）

### 输出
- 正向事件：商贸、学宫类即时收益（金币/科研/幸福）
- 负向事件：威胁触发的袭扰损失（粮食/金币/幸福）
- 事件日志（用于玩家感知与飞轮节奏提示）

### 关键规则（v0.x）
- 每小时结算执行一次事件判定：
  - 若 `EventCooldownHours > 0`，仅减少冷却，不触发新事件。
  - 若存在候选事件，基础触发概率  
    `P = clamp(0.12 + 候选事件数 × 0.08 + Threat × 0.0015, 0.12, 0.48)`
- 候选事件（按权重抽取）：
  - `商路集市`：需 `Merchants >= 10` 且 `Happiness >= 55`
  - `学宫讲习`：需 `Scholars >= 8`
  - `边境袭扰`：需 `Threat >= 42`
- 抽中后立刻写入资源与幸福变化，并写日志。

### 边界条件
- 资源扣减后必须保持非负（最低为 `0`）。
- 冷却小时数不可为负（每次结算后用 `max(..., 0)`）。
- 旧存档缺少 `EventCooldownHours` 时，默认 `0`，不影响读档。

## I. 县城地图生成（中式 2.5D）系统

### 目标
- 在“县城地图”页提供可重复生成的城镇布局，让地图层级切换具备可感知反馈。

### 输入
- `Population`（人口规模提示）
- `HousingCapacity`（居住容量提示）
- `ElitePopulation`（中高层建筑概率提示）
- `LayoutSeed`（布局种子，用于可复现生成）

### 输出
- 地块网格（`Ground / Road / Courtyard / Water`）
- 房屋集合（格位、朝向、层数、门楼标记）
- 2.5D 渲染结果（斜面地块 + 中式屋顶/墙体）

### 关键规则（v0.x）
- 道路骨架：先生成横向与纵向主街，再基于随机分支扩展支路，形成非规则街巷。
- 房屋目标数量：  
  `TargetHouses = clamp(round(Population × 0.22 + HousingCapacity × 0.05 + ElitePopulation × 0.6), 18, 72)`
- 房屋只允许落在道路邻接地块，避免“无路贴脸”与重叠占位。
- 2.5D 坐标投影：  
  `isoX = (gridX - gridY) × tileHalfWidth`  
  `isoY = (gridX + gridY) × tileHalfHeight`
- 建筑表现：基于房屋层数抬升墙体高度，并叠加蓝灰屋顶、暖色墙面与朱红点缀。

### 边界条件
- 地图生成失败时回退到空地网格，不得阻塞主 UI。
- 房屋数量不可超过可建地块数，且不得产生负索引/越界访问。
- 生成逻辑不得改写 `GameState` 经济与人口数值，仅使用提示输入进行可视化映射。

## H. 产业链岗位（建筑+工具约束）系统

### 目标
- 让岗位与产业基础设施强绑定：先有产业建筑与工具，再有对应岗位产出。

### 岗位映射（v0.x）
- `Farmer`：产业工人（农作/工坊一线）
- `Worker`：管理人员（官署/调度）
- `Merchant`：商业人员（市集/流通）
- `Scholar`：研发人员（学宫/研修）

### 输入
- 各岗位人数（四类）
- 建筑数量：农坊、工坊、学宫、市集、官署
- 工具库存：`IndustryTools`
- 资源库存：木、石、金

### 输出
- 岗位容量（受建筑限制）
- 工具覆盖率（受工具库存限制）
- 产业产出倍率（受管理覆盖和工具覆盖影响）
- 建筑建造日志、工具缺口日志

### 关键规则（v0.x）
- 岗位容量：
  - 产业工人容量 = `农坊×16 + 工坊×12`
  - 研发容量 = `学宫×10`
  - 商业容量 = `市集×14`
  - 管理容量 = `官署×8`
- 管理加成：  
  `ManagementBoost = 1 + clamp(管理人员 / 总人口, 0, 0.28)`
- 工具覆盖：  
  `ToolCoverage = clamp(IndustryTools / RequiredTools, 0.25, 1.0)`  
  `RequiredTools = 产业工人×0.34 + 研发×0.62 + 商业×0.42`
- 经济结算使用：  
  `有效产出 = 基础产出 × 管理加成 × 工具覆盖 × 科技倍率`

### 建筑与工具演化
- 每小时先执行产业系统：
  - 仅执行岗位容量校验与工具损耗，不自动扩建建筑
  - 若工具覆盖率过低，写入紧缺日志
- 玩家主动操作（底栏按钮）：
  - 可手动建造：农坊、工坊、学宫、市集、官署
  - 可手动制工具：消耗木石金并增加 `IndustryTools`
- 建造与制工具都要求存在管理岗位（`Worker > 0`），强化“组织能力”约束

### 边界条件
- 任一建筑数量最低为 `0`，不可出现负值。
- 工具库存最低为 `0`，不可出现负值。
- 若旧存档缺少建筑/工具字段，按默认初值读入并可继续结算。

## J. 主界面交互编排（双布局）

### 目标
- 保证主界面所有可见按钮都具备明确玩法行为，不出现“可点无效”。

### 输入
- UI 事件：探险开关、倍速切换、存档/读档/重置、产业扩建、制工具
- 当前布局模式：旧布局 / Figma 布局

### 输出
- 游戏状态变化（`GameState`）
- 事件日志反馈（`EventBus.LogAdded`）
- 主界面文字/数值刷新

### 关键规则（v0.x）
- 两套布局共享同一 `GameLoop` 与 `GameState`，仅节点绑定路径不同。
- 倍速规则：默认 `x1`，可切换 `x2`，通过 `GameLoop.SetTimeScale()` 控制分钟推进速度。
- 无论倍率如何，“每 60 游戏分钟结算 1 次”规则不变。
- Figma 中央动作按钮映射：
  - 左：扩建农坊
  - 中：扩建工坊
  - 右：锻造工具

### 边界条件
- 任一按钮触发失败时必须写入日志，且不得中断主循环。
- 布局切换仅影响显示与输入层，不得直接改写存档结构。
