# CountyIdle 系统规格

> 本文是 [01_game_design_guide.md](/E:/2_Personal/Finally/docs/01_game_design_guide.md) 的下位法。  
> `01` 裁决“这款游戏为何成立、主线往哪里收、哪些方向一律不收”；本文裁决“当前版本的系统如何落地、如何结算、如何观察、如何守边界”。  
> 若本文与 `01` 有冲突，**一律以 `01` 为准**。  
> 若本文与运行时代码冲突，应以“当前实现 + 已验证行为”为准，并回写本文修正。

## 0. 文档地位与使用裁定

- 本文只记录当前版本的核心系统法则，不承担灵感记录、草案堆放或实现备忘的职责。
- 任何系统若要进入主线，必须先回答它如何服务 `01` 中的三大体验支柱：
  - 宗门运营
  - 弟子调度
  - 挂机自动运转
- 任何系统若不能落在 `01` 中的三层空间之一，就不应写成主规格：
  - 山门内部
  - 外域周边
  - 长期成长
- 任何细到“面板结构、数值常量、世界生成权重、存档字段迁移”的内容，若不影响主法则，都不应反客为主挤占正文。

## 1. 项目基线与玩家可见语义

### 1.1 项目基线

- 项目：`CountyIdle`
- 引擎：`Godot 4.2+`
- 语言：`C# + .NET 8`
- 当前玩家身份：浮云宗掌门 / 宗主
- 当前经营主视角：天衍峰
- 当前组织骨架：天衍峰为主视角，青云峰三大总殿为总协同中枢

### 1.2 世界观与术语优先级

- 世界观锚点、组织结构、术语门禁，以 [09_xianxia_sect_setting.md](/E:/2_Personal/Finally/docs/09_xianxia_sect_setting.md) 为准。
- 玩家可见语义必须统一为：
  - `浮云宗`
  - `青云峰`
  - `天衍峰`
  - `阵堂 / 庶务殿 / 传法院 / 青云总坊接口`
  - `门人 / 弟子 / 附庸 / 职司 / 传承 / 护山 / 外务`
- `County / Town / Prefecture` 等旧词只允许作为代码层兼容命名存在，不代表玩家可见设定。

### 1.3 当前玩家可见语义映射

| 技术名 | 对外语义 | 当前裁定 |
| --- | --- | --- |
| `Population` | 门人总量 / 宗门人口池 | 不是城民人口，而是浮云宗生态中的可调配人口总量 |
| `Farmers / Workers / Merchants / Scholars` | 阵材 / 阵务 / 总坊外事 / 推演职司汇总 | 当前为内部结算汇总层，不作为玩家主交互对象 |
| `ContributionPoints` | 宗门贡献点 | 宗门内部调度与建设货币 |
| `Research` | 传承研修 / 功法技艺推演 | 不是现代科技研发 |
| `Breeding` | 灵根苗子培养 / 后辈培育 | 属于长期成长系统 |
| `Combat` | 外务历练 / 护山战备 | 同时覆盖外域与宗门防务语义 |
| `CountyEvent` | 宗门见闻 / 山门事件 | 属于宗门活态反馈，不是通用随机事件 |

## 2. 核心主轴与系统总法则

### 2.1 核心主轴

当前版本的一切系统都必须服务于以下主轴：

1. 经营山门，立住资源、堂口、人口与秩序基础
2. 收人、育人、分人，让宗门真正拥有可用之人
3. 指派门人去修炼、历练、采炼、行商、守山、办差
4. 门人的行动带回收益、损耗、成长、事件与风险
5. 玩家依据结果重排宗门结构、开发新地、改换治峰方向
6. 宗门因此获得更强的自转能力与更大的外伸空间

任何绕开这条主轴单独求爽的系统，只能是侧枝，不得作为本文核心规格。

### 2.2 三条系统总法

- 系统必须让宗门更像一个活的组织，而不是更像一张表。
- 系统必须让玩家能读懂“为何得此果”，而不是只看见跳动结果。
- 系统必须让手动裁决与挂机运转能自然接力，而不是彼此打断。

### 2.3 新系统准入条件

任何新系统进入本文前，至少要先满足以下四条：

- 它明确强化宗门运营、弟子调度或挂机自动运转中的至少一项
- 它明确落在山门内部、外域周边、长期成长中的至少一层
- 它能回流到现有飞轮，而不是形成孤岛玩法
- 它不破坏 `01` 中的伦理、题材与边界要求

## 3. 三层空间的系统落点

### 3.1 山门内部

山门内部是当前版本最优先、最稳定、最常驻的系统层。

当前应由以下系统承载：

- 堂口建设、升级与职能分化
- 库藏、资源链、法器与材料周转
- 门人驻留、修炼、生产、庶务、恢复
- 护山阵务、基础秩序与内务调拨
- 宗主中枢、法令、贡献点与内务治理

系统要求：

- 山门系统必须回答“天衍峰今日凭什么活、凭什么稳、凭什么能再往前一步”。
- 任何山门系统都不得退化成“纯建筑摆放小游戏”。
- 山门系统的结果必须能回写到门人状态、外域行动或长期成长中的至少一项。

### 3.2 外域周边

外域是宗门把峰内秩序伸向峰外的系统层。

当前应由以下系统承载：

- 原材来源与附庸供给
- 青云总坊接口、商路与外务回流
- 历练、采炼、行商、驻守与附庸往来
- 妖患、袭扰、威胁与扩张机会
- 宗门护持关系与外域影响范围

系统要求：

- 外域不是风景图，也不是资源井；它必须对宗门形成压力与回报的双向关系。
- 外域系统必须能把结果回流到山门库存、门人状态、威胁压力或长期成长。
- 不允许把凡俗势力、附庸据点写成可随意牺牲的“资源块”。

### 3.3 长期成长

长期成长是“宗门最终成为什么门派”的系统层。

当前应由以下系统承载：

- 传承研修与技艺积累
- 灵根苗子与后辈培养
- 真传 / 英雄单位的成长预留
- 宗门方向分化
- 护山体系与长期势力格局

系统要求：

- 长期成长不能只表现为总量变大，必须体现方向、代价、门风与层级差异。
- 长期成长必须与山门内部和外域周边形成双向耦合，而不是单独拉一条后期数值线。

## 4. 三大体验支柱的系统约束

### 4.1 宗门运营

宗门运营是整个系统群的地基。

当前版本必须保证：

- 玩家一直在经营宗门本身，而不是借壳去玩别的类型
- 建设、资源、库藏、治理、防务、外务之间存在互相喂养关系
- 贡献点与灵石承担不同职责，不得退化成一切都只花一种货币
- 宗门不是背景板，而是所有系统结果最终回流的对象

### 4.2 弟子调度

弟子调度是将纸面规划变成宗门行动的系统轴。

当前版本的裁定是：

- 玩家不以逐人逐秒微操为主
- 玩家也不能完全跳过“用什么人做什么事”的判断
- 当前运行版允许通过职司汇总层做兼容结算
- 但所有调度系统都必须朝“门人承担真实职责”这一方向收口

当前门人相关系统至少应覆盖：

- 人口池与基础分层
- 调度与任务承担
- 修炼 / 历练 / 办差 / 驻守的用途分化
- 长期培养对象的识别与回报

### 4.3 挂机自动运转

挂机是本项目成立条件之一，不是简化模式。

当前版本必须保证：

- 玩家下达的安排可持续执行
- 日常小波动尽量自行吸收
- 重大机会与重大风险必须留下清晰反馈
- 自动运转不能轻易把系统推入不可恢复死局
- 玩家重新接手时，能快速看懂“发生了什么、现在该治哪一步”

挂机的及格线不是“能代跑”，而是“能代宗门继续活下去”。

## 5. 时间与节奏系统

### 5.1 当前运行版硬基线

- 细时间按“现实时间持续推动游戏分钟”运行，具体换算口径由当前运行配置裁定
- 倍速范围：`x1 / x2 / x4`
- 每累计 `60` 游戏分钟触发一次小时结算
- 顶栏时间表达必须服务“宗门在动 + 岁月在压”的双重感受

### 5.2 双层时间制裁定

`01` 对时间的裁定是：

- `细时间` 负责让宗门活着
- `长时间` 负责让修仙世界有岁月

因此本文的系统约束为：

- 小时结算属于细时间层
- 日 / 月 / 季属于长时间反馈层
- 长时间只能建立在细时间累计之上，不能脱离现有运行状态另起一套世界
- 任何长时间系统若要进入当前实现，必须说明它如何从小时结算累计而来

### 5.3 会话节奏要求

当前版本的系统反馈必须支持以下会话链路：

1. 观局
2. 批令
3. 放行
4. 收报
5. 再断

因此 UI、日志、结算、事件都必须服务“让玩家判断下一手”，而不是只服务表现。

## 6. 当前核心系统群

### 6.1 治峰与内务系统

治峰与内务是当前版本最核心的主系统群。

它至少包括：

- 堂口建设与升级
- 宗主中枢与治理法旨
- 贡献点与内务流转
- 库藏与资源调拨
- 护山与秩序维持

当前裁定：

- 玩家主交互应收口到“方向、法令、法旨、取舍”层，而不是裸露的逐项人数拨杆
- 兼容层可以保留内部汇总字段，但不能回退玩家可见语义
- 内务系统必须始终与门人状态、外域行动和长期成长保持互相影响

### 6.2 门人与职司系统

当前门人系统以“宗门人口池 + 职司汇总 + 调度回流”构成最小闭环。

本文裁定：

- `Population` 对外只能解释为宗门人口池
- 当前允许存在 `Farmers / Workers / Merchants / Scholars` 作为内部兼容汇总
- 但玩家主交互应表达为职司、法旨、用途与培养方向
- 后续若实现转任、真传、英雄实体，应视作现有门人系统的显式化，而不是另起新系统

### 6.3 产业、供养与库藏系统

产业系统不只是产能系统，它是宗门供养能力的体现。

本文裁定：

- 资源必须支撑建造、供养、防务、历练与长期成长
- 材料链应保持“原材 -> 加工 -> 消费 / 反哺”的可解释结构
- 仓储系统必须承担真实压力，不能只是上限装饰
- 产业系统的结果必须能解释给玩家听，而不是只在后台结算

当前材料层级方向：

- `T0 / T1` 作为当前版本主要前台层级
- 更高层材料与复杂制造可继续存在，但不得挤占主线可读性

### 6.4 传承与长期成长系统

传承系统属于长期成长主轴，而不是附属数值条。

本文裁定：

- 研修系统对外统一解释为传承 / 功法技艺推演
- 传承必须与堂口、门人、外务、护山形成联动
- 灵根苗子培养属于长期成长，不得写成纯概率抽奖系统
- 真传 / 英雄系统若进入实现，必须服务宗门史与组织感，而不是单角色爽感

### 6.5 外务、历练与护山系统

外务系统负责把山门与外域接起来。

本文裁定：

- 历练不是纯战斗玩法，而是收益、损耗、成长与风险的外域回路
- 护山不是单一防御玩法，而是宗门秩序、威胁压力与长期护持责任的体现
- 外务系统必须与资源、门人、传承、威胁形成闭环
- 不允许把外务系统做成脱离宗门经营的纯关卡制玩法

### 6.6 装备与法器系统

装备 / 法器系统只能作为门人、历练、供养链的下位支撑。

本文裁定：

- 装备系统不能喧宾夺主，抢走宗门经营与门人调度的主位
- 其价值必须回流到历练、生存、防务、生产效率或培养方向
- 若当前实现以掉落和工具制作为主，也应保持未来可向法器体系平滑过渡

### 6.7 事件系统

事件系统必须服务“宗门是活的”这一核心判断。

本文裁定：

- 事件必须来自山门、外域、传承、护山、门人状态等真实系统来源
- 事件不是独立的随机噪音
- 事件至少要提供以下三者之一：
  - 结果反馈
  - 风险警示
  - 新的判断空间

## 7. 结算顺序、状态安全与可观测性

### 7.1 当前小时结算顺序

当前运行版固定顺序为：

`Industry -> Resource -> Economy -> Research -> Population -> Breeding -> Combat -> CountyEvent`

裁定：

- 这是当前主结算链，不得随意交换
- 任何新子系统若要插入，必须证明它不会打断主链可读性与稳定性
- 小时结算完成后，UI 读取的必须是 `GameState.Clone()` 发布结果

### 7.2 GameState 安全约束

以下条目属于“破了就算 bug”的硬边界：

- 人口、岗位、建筑数量、工具库存不得出现负值
- 资源扣减路径必须有可支付检查
- 玩家可见库存必须保持可解释
- 新字段必须保证旧存档可读
- 自动运转不得轻易把系统推入不可恢复死局

### 7.3 可观测性要求

当前系统至少要能向玩家解释：

- 这一轮宗门做成了什么
- 哪条链路开始发紧
- 哪一类风险正在抬头
- 现在最值得宗主处理的是什么

因此日志、摘要、状态提示至少要覆盖：

- 资源与供养压力
- 门人与调度结果
- 外务与威胁变化
- 长期成长推进

## 8. 地图、界面与存档的定位

### 8.1 地图定位

地图是三层空间的观察与调度载体，不是独立玩法主轴。

本文裁定：

- 宗门图服务山门内部
- 外域图 / 世界图服务外域周边
- 地图系统的职责是观察、进入、调度、反馈
- 地图不得沦为纯观光图，也不得脱离宗门飞轮自成一套玩法

#### 8.1.1 天衍峰院域坊局与全格检视（一期）

本文裁定：

- 天衍峰山门图的任意 hex 都必须可点选；若未命中场所锚点，则进入“院域坊局检视”分支，不再退回纯背景格。
- 每个院域地块至少维护以下固定底盘字段：`RegionName`、`ContentKind`、`QiAffinityText`、`BaseQiCapacity`、`QiRecoveryPerHour`、`BuildSlotCount`、`FeatureTexts`。
- 每个院域地块的一期坊局层至少维护以下可变字段：`PlanStyle`、`SubBuildings`、`TotalQiDemand`、`QiCongestion`、`SynergyScore`、`Stability`、`SuggestedBuildType`。
- 检视器一期必须统一回答四个问题：`这里是什么`、`它现在在干什么`、`为什么顺或为什么卡`、`我现在能做什么`。
- 当前运行版至少支持三档运行时坊局调整：`主修坊局`、`协同坊局`、`稳态坊局`；切换后要立即刷新坊位摘要、灵气需求、协同分和稳定度。
- 天衍峰山门图与同族二级局部沙盘的基础地块层，必须优先直接读取 `L1_hex_tileset.tres` 作为运行时纹理来源；不允许长期维持“编辑器已有 tileset、运行时仍直接切图片 atlas”的双轨漂移状态。
- 若基础地块 atlas 本身为方形切片，运行时必须按 hex polygon 进行投影，保证玩家看到的是无缝 hex 地块，而不是原始矩形贴图区。
- 一期允许把坊局切换停留在地图局部模拟层，不强制立刻写入 `GameState` 小时结算，也不强制立刻改动存档结构；未接入结算前，相关数值视为院域规划反馈，不视为全局经济事实。
- 院域随机性来源继续限定为：`地块 traits`、`节气波动`、`低频局部事件`、`驻守差异`、`宗门缺口`；一期只允许先接入说明与规划反馈，不允许用高频随机把玩家长期规划直接打散。

#### 8.1.2 世界格二级地图入口（一期）

本文裁定：

- 世界地图的任意 hex 都必须具备“进入下一层地图”的身份语义；不允许只有少数世界站点可进入，而其余地块只是不可交互背景。
- 世界图点击链路采用 `站点优先、地块回退`：若点击命中已生成站点，则沿用该站点的 `PrimaryType / SecondaryTag / RegionId / RarityTier / UnlockTier`；若未命中站点，则必须基于当前地块的 `Biome / Terrain / Water / Wonder / Structure / QiDensity / Corruption / MonsterThreat / Fertility` 合成一份可进入的二级地图入口数据。
- 站点回退生成的一期类型至少覆盖 `Wilderness`，并允许继续映射到 `Sect / MortalRealm / Market / CultivatorClan / ImmortalCity / Ruin`；其目标不是一次性补完独立玩法，而是保证“每格可进、每格有身份、每格有风险/收益描述”。
- 点击世界 hex 后，运行时最小闭环为：`世界图点选 -> 左侧检视刷新 -> 左侧进入按钮 -> SecondaryMapView 生成下一层局部地图 -> 可返回世界图`。
- `SecondaryMapView` 一期仍可复用统一占位页，但必须能按 `PrimaryType` 显示差异化的 `核心玩法 / 主要产出 / 主要风险` 文案；`Wilderness` 不得回落到默认未分类占位文案。
- `SecondaryMapView` 在当前运行版必须包含一张按所选 world hex 语义生成的局部地图；地图生成至少要参考该格的 `Biome / Terrain / Water / Wonder / Structure / QiDensity / Corruption / MonsterThreat / Fertility` 中的可用字段。
- 上述局部地图在表现形态上必须与天衍峰山门沙盘保持同族：继续使用同类 hex 沙盘视图、缩放/点选节奏与地块检视习惯，只允许改变地图内容、地块语义和局部布局，不另起一套完全不同的预览样式。
- 世界图一期的基础地块层当前继续复用 `L1_hex_tileset.tres`，但运行时主链已回到 `StrategicMapViewSystem._Draw()` 按 hex polygon 逐格投 atlas 区域；此前直接把世界格写入 `WorldTerrainTileLayer` 的方片排布会在六边形裁切后留下连续白缝，因此该节点目前仅保留为后续实验/备用基础设施，不再承担正式世界图底盘。道路 / 河流 / 站点 / 标签 / 选中高亮仍继续由 `StrategicMapViewSystem` 脚本叠加层承接，避免把交互反馈压进纯地块层后失真；旧版蜂窝背景网格也不再叠加。
- 当前阶段允许把“世界格二级地图”停留在模板化入口层，不强制立即接入专属场景、独立结算或存档结构；但所有世界格都必须能给出下一层身份、开放层级与回流方向。

### 8.2 界面定位

界面的职责是组织判断，不是承载复杂玩法计算。

本文裁定：

- `Main.cs` 继续只负责组织、切换与展示
- 复杂玩法计算应留在系统层
- 弹窗、面板、图册都必须服务“观局 -> 批令 -> 放行 -> 收报 -> 再断”

### 8.3 存档定位

存档是运行稳定性的基础设施，不是当前主设计主轴。

本文裁定：

- 存档与读档必须服务当前宗门连续经营体验
- 存档格式与槽位能力属于运行支撑层
- 存档细节不得反客为主挤占系统主法则

## 9. 设计边界与禁止项

当前版本一律不以以下方向作为主规格中心：

- 单角色重操作 ARPG
- 纯建筑铺设、弱化门人调度的城建玩法
- 只追求效率表格、不追求宗门组织感的算表玩法
- 只追求视觉包装、不追求系统反馈可解释性的展示玩法

任何系统若出现以下倾向，应直接退回：

- 为复杂而复杂
- 强迫逐人逐秒微操
- 把门人、附庸、据点写成纯耗材
- 让题材只停留在换词层面
