# 改动提案（Change Proposal）

## 提案信息

- 标题：岗位面板进程化交互与具体岗位名展示
- 日期：2026-03-07
- 提案人：Codex
- 变更级别：`L2 机制`

## 改动背景

- 当前问题：`JobsPadding` 中每条岗位行只有抽象职业统称，数值来源与建筑/科技关系不可解释，且行本身缺少查看规则的直接交互。
- 证据（数据/玩家反馈）：用户反馈“JobsPadding 里面每个元素都能够有交互，并且其具体的数值需要有一个规则进行管控，这里面的岗位也需要和建筑、科技进程相对应，需要展现出实际的具体岗位名称，而不是一个笼统的名称”。

## 改动内容

- 改什么：
  - 为四条岗位行增加展开/收起交互，用于查看岗位规则详情
  - 新增岗位进程规则，按建筑数量与科技层级映射具体岗位名称
  - 在岗位详情与岗位调整日志中展示容量公式、建筑快照、解锁条件
- 不改什么：
  - 不把四大职业拆成独立子职业数值池
  - 不改动小时结算频率与核心飞轮顺序
  - 不引入完整的 `data/jobs.json` 配置化加载
- 影响系统：`docs/02_system_specs.md`、`CountyIdle/scripts/Main.cs`、`CountyIdle/scripts/systems/JobProgressionRules.cs`、`CountyIdle/scripts/core/GameLoop.cs`、`CountyIdle/scenes/ui/JobRow.tscn`

## 预期结果

- 预期提升指标：
  - `JobsPadding` 可见岗位行“无直接解释”的问题降至 0
  - 岗位容量来源在面板内可直接读出
  - 建筑/科技推进会在岗位标题上产生即时可见反馈
- 可接受副作用：
  - 当前仍然只有四类岗位数值池，具体岗位名称主要承担表现与解释职责

## 验证计划

- 验证方式：
  - `dotnet build .\Finally.sln`
  - Godot 打开 Legacy 布局，点击岗位行查看展开/收起与文本变化
- 观察周期：单次会话（5~10 分钟）
- 成功判定阈值：四条岗位行均可交互、标题与详情可正确响应建筑/科技变化，且无运行时异常

## 回滚条件

- 触发条件：岗位面板交互导致按钮失效、布局错乱、日志异常或存读档异常。
- 回滚步骤：回退 `Main.cs`、`JobRow.tscn` 与 `JobProgressionRules.cs` 本次改动并恢复固定岗位标题。
