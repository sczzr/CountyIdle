# 改动提案（Change Proposal）

## 提案信息

- 标题：岗位与产业基础设施强绑定改造
- 日期：2026-03-06
- 提案人：Gameplay Agent
- 变更级别：`L2 机制`

## 改动背景

- 当前问题：岗位可直接分配且即时产出，缺少“产业基础条件”，不符合生产关系演化逻辑。
- 证据（数据/玩家反馈）：现有岗位仅受总人口限制，未受建筑与工具约束，导致岗位抽象化过强。

## 改动内容

- 改什么：
  - 四岗位重定义：产业工人/管理人员/商业人员/研发人员
  - 引入建筑容量与工具覆盖率，岗位上限与产出效率由产业基础决定
  - 引入每小时产业运维（容量校验 + 工具损耗）
  - 新增玩家主动产业操作（手动建造建筑 + 手动制工具）
  - 经济结算加入管理加成与工具覆盖率
- 不改什么：
  - 不新增复杂建筑面板（仅底栏快捷操作）
  - 不改探险战斗机制
  - 不改存档文件路径与读写流程
- 影响系统：
  - `scripts/models/GameState.cs`
  - `scripts/systems/IndustryRules.cs`（新增）
  - `scripts/systems/IndustrySystem.cs`（新增）
  - `scripts/systems/EconomySystem.cs`
  - `scripts/core/GameLoop.cs`
  - `scripts/Main.cs`
  - `scenes/ui/JobsPanel.tscn`

## 预期结果

- 预期提升指标：
  - 10 分钟内能在日志中观察到“建造/工具/产业状态”反馈
  - 岗位分配决策从“只看人数”升级为“看产业条件+人数”
  - 管理岗位从弱感知岗位变为关键加成位
  - 玩家可主动决定“先扩建哪条产业线”，策略感更强
- 可接受副作用：
  - 初期节奏略慢，需用管理岗位与建造节奏换取稳定增长

## 验证计划

- 验证方式：
  - `dotnet build .\Finally.sln`
  - 运行观察岗位加点受容量约束与日志反馈
  - 观察工具不足时资源增速下降但系统不崩
  - 存档/读档后建筑与工具状态保持
- 观察周期：10~20 分钟挂机窗口
- 成功判定阈值：无负值异常、无崩溃、岗位逻辑符合“先产业后岗位”

## 回滚条件

- 触发条件：资源收缩过强导致飞轮中断（连续多个小时无有效增长）
- 回滚步骤：
  1. 移除 `IndustrySystem` 调度
  2. 还原 `EconomySystem` 为岗位线性产出
  3. 保留文档并记录失败原因
