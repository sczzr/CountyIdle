# 改动提案（Change Proposal）

## 提案信息

- 标题：天衍峰院域坊局与全格检视（一期）
- 日期：2026-03-11
- 提案人：Codex
- 变更级别：`L2 机制`

## 改动背景

- 当前问题：天衍峰山门图目前更偏表现层底图，地块缺少稳定的可检视语义，玩家也无法围绕单地块做“多建筑配套 + 灵气分配 + 局部诊断”的经营决策。
- 证据（数据/玩家反馈）：本轮讨论里，用户明确提出“地块固定但可容纳多个建筑”“地块灵气总量固定且需共享”“同地块建筑要能形成产业 buff”“同时要有随机性，避免最优模板固化”。

## 改动内容

- 改什么：
  - 将天衍峰单地块设计为“固定外层 hex + 内层坊局组合”的双层经营对象；
  - 引入共享灵气池、坊局协同、局部随机性与全格点击检视；
  - 一期先完成中文设计规格、功能包登记与数据层边界，不一次性压上完整编辑器。
- 不改什么：
  - 不在本提案阶段直接改动小时结算公式；
  - 不立即调整 `GameState` 存档结构；
  - 不恢复弟子实体可视移动。
- 影响系统：
  - `SectMap`、`TownMapGeneratorSystem`、`CountyTownMapViewSystem`、`Tile Inspector`
  - `docs/02_system_specs.md`
  - `docs/05_feature_inventory.md`
  - `docs/08_development_list.md`

## 预期结果

- 预期提升指标：
  - 山门图点击后具备明确的信息价值和决策价值；
  - 地块布局从“看背景”升级为“做局部配套”；
  - 后续建筑链、弟子驻守、灵脉、事件、护山都能自然挂到同一格数据模型上。
- 可接受副作用：
  - 一期只是写清设计与数据骨架，短期内仍会出现“有设计、功能未完全做完”的阶段性落差；
  - 若坊局层规则过多，后续需要再压缩字段和按钮数量。

## 验证计划

- 验证方式：
  - 检查 `02 / 05 / 08 / FC / CP` 是否对齐；
  - 后续实现阶段，验证任意地块点击是否能展示灵气、坊位、天然特征、协同提示与推荐动作。
- 观察周期：文档阶段当次完成；实现阶段按单迭代验证。
- 成功判定阈值：
  - 设计文档中已明确“地块固定底盘 / 坊局可变组合 / 灵气共享预算 / 随机性来源”四个核心概念；
  - 后续实现不会再回退为“仅场所锚点可点”的旧模型。

## 回滚条件

- 触发条件：
  - 若该方案导致地块规则复杂度过高、玩家 10 分钟内无法感知收益，或实现成本明显超出当前迭代承载范围。
- 回滚步骤：
  - 回退到“全格点击检视 + 单地块固定语义”；
  - 保留字段骨架，暂缓坊局组合、协同和随机性扩展。
