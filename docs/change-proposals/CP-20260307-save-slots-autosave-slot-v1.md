# 改动提案（Change Proposal）

## 提案信息

- 标题：自动存档槽策略（V1：单自动槽）
- 日期：2026-03-07
- 提案人：Gameplay Agent
- 变更级别：`L2 机制`

## 改动背景

- 当前问题：
  - 多存档 UI 已支持手动槽，但仍缺少自动存档保护层；
  - 玩家如果忘记手动存档，仍可能损失最近阶段的经营进度；
  - 当前只有 `default` 主存档与手动槽，缺少自动恢复兜底。
- 证据（数据/玩家反馈）：
  - 看板中“多存档槽管理界面”条目尚包含“自动存档槽策略”缺口；
  - 用户已明确要求继续推进存档体系。

## 改动内容

- 改什么：
  - 新增 `autosave / 自动存档` 受保护槽位
  - 每 `6` 次小时结算自动写入一次当前 `GameState`
  - 多槽面板展示自动槽，但禁止手动覆盖、重命名、删除
- 不改什么：
  - 不实现自动存档轮换
  - 不实现退出时自动存档
  - 不改变快速存档 / 快速读档的主存档语义
- 影响系统：
  - `CountyIdle/scripts/core/SaveSystem.cs`
  - `CountyIdle/scripts/ui/SaveSlotsPanel.cs`
  - `CountyIdle/scripts/ui/MainSaveSlotsPanel.cs`
  - `CountyIdle/scripts/Main.cs`
  - `docs/02_system_specs.md`

## 预期结果

- 预期提升指标：
  - 降低未手动存档时的进度损失风险
  - 多槽面板更完整地体现 SQLite 存档价值
  - 维持默认槽、自动槽、手动槽三种职责清晰分工
- 可接受副作用：
  - 每隔若干小时结算会多一次数据库写入

## 验证计划

- 验证方式：
  - 推进游戏至 `HourSettlements` 达到 `6` 的倍数
  - 打开多槽面板检查 `自动存档` 是否出现
  - 验证自动槽可读但不可改名、不可删除、不可手动覆盖
  - `dotnet build .\Finally.sln`
- 观察周期：一轮启动 + 至少一次自动存档触发
- 成功判定阈值：
  - 自动槽按预期写入
  - 不会在读档后立即错误覆盖
  - 不破坏现有手动槽流程

## 回滚条件

- 触发条件：
  - 自动存档触发过频
  - 自动槽保护逻辑失效
  - 读档后立即异常覆盖自动槽
- 回滚步骤：
  1. 保留自动槽展示
  2. 关闭自动写入触发逻辑
  3. 保持主存档与手动槽流程不变
