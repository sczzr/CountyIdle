# 改动提案（Change Proposal）

## 提案信息

- 标题：自动存档轮换槽（V1：保留最近 3 个自动档）
- 日期：2026-03-08
- 提案人：Gameplay Agent
- 变更级别：`L2 机制`

## 改动背景

- 当前问题：
  - 单自动存档槽虽然能兜底，但只能保留最近一次自动记录；
  - 若玩家在自动存档后继续操作并出现误判，无法回退到更早的自动状态；
  - 多存档 UI 已具备展示多个槽位的能力，适合继续推进为轮换自动档。
- 证据（数据/玩家反馈）：
  - 上一轮交付建议的下一步已明确指向“自动存档轮换槽”；
  - 当前面板已能承载受保护自动槽，不需要重新设计交互容器。

## 改动内容

- 改什么：
  - 自动存档槽从 1 个升级为 3 个轮换槽
  - 自动存档规则保持“每 `6` 次小时结算触发一次”
  - 轮换顺序按 `自动存档 1 -> 自动存档 2 -> 自动存档 3 -> 自动存档 1` 循环
- 不改什么：
  - 不改变主存档 `default`
  - 不改变手动槽逻辑
  - 不改变 `QuickSave / QuickLoad` 的默认槽语义
- 影响系统：
  - `CountyIdle/scripts/core/SaveSystem.cs`
  - `CountyIdle/scripts/ui/MainSaveSlotsPanel.cs`
  - `CountyIdle/scripts/ui/SaveSlotsPanel.cs`
  - `docs/02_system_specs.md`

## 预期结果

- 预期提升指标：
  - 自动存档的回退余量从 `1` 次提升到 `3` 次
  - 不增加新的 UI 路径复杂度
  - 自动槽保护逻辑仍然简单清晰
- 可接受副作用：
  - SQLite 写入槽位数增加，但频率不变

## 验证计划

- 验证方式：
  - 模拟至少 `18` 次小时结算，观察自动槽是否依次轮换
  - 打开多槽面板，确认自动存档 1/2/3 均存在并受保护
  - `dotnet build .\Finally.sln`
- 观察周期：单轮启动至自动存档轮换满 3 次
- 成功判定阈值：
  - 自动槽轮换顺序正确
  - 面板展示与保护无误

## 回滚条件

- 触发条件：
  - 自动槽顺序异常
  - 同一槽被连续误覆盖
  - 保护逻辑对自动槽失效
- 回滚步骤：
  1. 保留多槽 UI
  2. 将自动存档槽回退为单槽
  3. 保留默认槽与手动槽逻辑
