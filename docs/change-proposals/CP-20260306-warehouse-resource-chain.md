# 改动提案（Change Proposal）

## 提案信息

- 标题：仓储容量与矿产资源链机制接入
- 日期：2026-03-06
- 提案人：Gameplay Agent
- 变更级别：`L2 机制`

## 改动背景

- 当前问题：现有资源体系以粮木石金为主，缺少矿产开采、材料研发与产品制造链路；同时缺少仓储容量约束，资源无上限堆积。
- 证据（数据/玩家反馈）：新增需求明确提出“仓储 + 矿石 + 开采 + 研发新材料 + 产品制造 + 建筑制造”的完整使用场景。

## 改动内容

- 改什么：
  - 新增仓储容量系统（仓储等级、容量计算、超容溢出）
  - 新增矿产链路：铁矿/铜矿/煤矿开采 → 金属锭冶炼
  - 新增材料研发链路：金属锭 + 科研 → 复合材料
  - 新增制造链路：新材料转化为工业部件与建材
  - 建筑建造与制工具接入新链路消耗
  - 新增“矿仓联建”操作，联动矿坑等级与仓储等级
- 不改什么：
  - 不新增独立仓库界面
  - 不改探险战斗胜负公式
  - 不改存档路径与结算基线（1s=1分钟、60分钟结算）
- 影响系统：
  - `CountyIdle/scripts/models/GameState.cs`
  - `CountyIdle/scripts/systems/IndustryRules.cs`
  - `CountyIdle/scripts/systems/IndustrySystem.cs`
  - `CountyIdle/scripts/systems/ResourceSystem.cs`（新增）
  - `CountyIdle/scripts/core/GameLoop.cs`
  - `CountyIdle/scripts/Main.cs`
  - `docs/02_system_specs.md`

## 预期结果

- 预期提升指标：
  - 资源循环从“基础产出”升级为“基础产出 + 矿产深加工”
  - 仓储容量成为中期发展约束，推动矿仓联建决策
  - 新材料可同时反哺产品制造与建筑建造，形成多场景消耗
- 可接受副作用：
  - 资源管理复杂度上升，短期学习成本提高

## 验证计划

- 验证方式：
  - `dotnet build .\Finally.sln`
  - 运行观察小时结算日志：开采、冶炼、研发、制造、溢出
  - 验证矿仓联建按钮可触发并更新状态
  - 验证建造/制工具的新增消耗路径
- 观察周期：10~20 分钟挂机窗口
- 成功判定阈值：无负值异常、无结算中断、仓储约束与资源链日志可解释

## 回滚条件

- 触发条件：资源链导致核心飞轮停滞（连续多个小时无法维持基础增长）
- 回滚步骤：
  1. 从 `GameLoop` 移除 `ResourceSystem` 调度
  2. 还原 `IndustrySystem` 的建造/制工具新增资源消耗
  3. 保留文档并记录失败原因
